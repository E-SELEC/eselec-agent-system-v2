# Regla de admision de canon

Usar esta regla antes de crear, migrar, dividir, resumir o reforzar cualquier `*-canon` del sistema E-SELEC v2.

## Principio rector

Un canon no es una skill larga.

Un canon es una puerta de criterio transversal que cambia como piensa un area completa. Si no tiene profundidad suficiente, debe ser una skill, referencia, checklist, template o planning note, pero no canon.

## Estandar SEO

El canon SEO es el estandar minimo de comparacion porque nacio de un proceso formativo real:

- fuente primaria verificable;
- correcciones y reglas de Rodrigo;
- modulos cerrados;
- criterio para web nueva y web existente;
- procesos paso a paso;
- checklists;
- errores comunes;
- limites para perfiles junior;
- fuentes y links;
- regla final de relevo;
- indice de lectura bajo demanda;
- declaracion obligatoria de uso.

No crear otro canon si no puede aspirar a ese mismo nivel de exigencia.

## Condiciones minimas

Un canon futuro debe cumplir todo esto antes de aceptarse:

1. **Fuente primaria verificable.** Debe venir de PDFs, sesiones de formacion directa, documentacion oficial, pruebas reales o conversaciones fuente entregadas por Rodrigo. Un resumen, handoff o auditoria no basta por si solo.
2. **Densidad modular.** Debe tener varios modulos sustantivos. Como regla practica, si no hay al menos 10 piezas de aprendizaje o una excepcion muy justificada, todavia no es canon.
3. **Estructura interna repetible.** Cada modulo debe poder cubrir: definicion, idea simple, regla dura, web nueva vs web existente cuando aplique, proceso, checklist, errores comunes, que NO debe hacer un junior, fuentes y regla de relevo.
4. **Indice propio.** Debe tener un indice equivalente a `indice-canon-seo.md`: archivo, funcion, cuando leerlo, quien lo usa y que problema resuelve.
5. **Lectura bajo demanda.** El `SKILL.md` del canon debe mapear `caso -> archivos a leer`. No se debe cargar todo por costumbre.
6. **Declaracion de uso.** El agente debe poder responder: que canon leyo, por que lo leyo y que decision cambio. Si no cambio ninguna decision, no aplico canon.
7. **Separacion fuente/canon.** Las fuentes historicas explican origen. El canon operativo contiene criterio utilizable. No mezclar archivos fuente, memoria de cliente e instrucciones vivas.
8. **Criterio de activacion.** El frontmatter `description` debe decir claramente cuando se activa. No puede ser generico ni aspiracional.
9. **No contaminacion de clientes.** Casos reales solo entran si son anonimizados como patron reusable. Los datos de clientes viven en `clients/[cliente]/`.
10. **Prueba de calidad.** Debe existir forma de comprobar que el canon mejora decisiones: checklist, contraste con output anterior, evaluacion por Rodrigo o auditoria de alineacion.

## Decision por defecto

Si falta cualquiera de las condiciones anteriores:

- no crear canon;
- reforzar la skill existente;
- crear referencia bajo demanda;
- crear checklist o template;
- documentar el aprendizaje en `planning/`;
- observar outputs reales antes de escalar.

## Prohibiciones

- No crear canon por analogia con SEO.
- No convertir un resumen de buenas practicas en canon.
- No usar clientes reales como instrucciones generales.
- No reemplazar PDFs, conversaciones fuente o documentacion oficial por una reinterpretacion corta.
- No afirmar que un canon existe si solo hay un `SKILL.md` sin indice ni referencias.

## Cierre obligatorio

Toda propuesta de canon debe cerrar con:

```text
CANON PROPUESTO:
Fuente primaria:
Modulos:
Indice:
Casos de activacion:
Decision que mejora:
Riesgo de contaminacion:
Alternativa si no se aprueba:
```
